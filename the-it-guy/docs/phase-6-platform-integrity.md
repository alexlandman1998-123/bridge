# Phase 6 Platform Integrity Notes

## Validation Engine

Phase 6 adds a central validation layer under `src/services/validation`.

The validation engine checks:

- user profile integrity
- app role validity
- membership validity
- workspace ownership/settings/default structures
- attorney firm compatibility
- onboarding completion contracts
- transaction ownership/stage consistency
- orphaned memberships and branches

Validation does not silently repair production data. It returns issues, severity, and repair recommendations.

## Repair Model

Repair suggestions are generated by `repairSuggestions.js`.

Supported recommendations include:

- recreate profile
- relink workspace
- create or reactivate membership
- create default branch
- repair workspace settings
- repair onboarding state
- normalize legacy role
- relink transaction
- manual review

Repairs are intentionally recommendations only. Production repair execution should require platform-admin review and should be logged in `repair_logs`.

## State Transitions

`stateTransitionEngine.js` defines canonical status transitions for memberships, onboarding, and transaction stages.

Blocked examples:

- `removed -> active`
- `signup_started -> onboarding_completed`
- transaction stage updates without required participants/documents/approvals

## Diagnostic Tooling

`/platform/diagnostics` is a platform-admin-only route for:

- user validation
- workspace validation
- transaction validation
- system integrity checks
- repair recommendation review

## Migration

`202605240006_platform_integrity_foundation.sql` adds:

- `integrity_logs`
- `repair_logs`
- `validation_states`
- `system_health_snapshots`

These tables are RLS-protected for `platform_admin` profiles.

## Production Rules

Production must not:

- silently repair corrupted records
- hide validation failures
- allow invalid memberships into dashboards
- allow invalid transitions
- expose raw database errors
- use localStorage or frontend assumptions as integrity authority

## Phase 7 Risks

- Some critical writes still need deeper service-specific guards.
- Transaction stage requirements need richer document/approval checks per workflow lane.
- RLS policies should mirror validation contracts at the database boundary.
- Repair execution workflows should be added after support/admin approval rules are finalized.
