# Transaction Workflow Canonical Contract v1

**Date:** 2026-06-01  
**Owner:** Transaction Workspace Product + Delivery  
**Scope:** Phase 1 definition only (no behavior refactor yet).  
**Goal:** Define one canonical contract so the parent overview is derived from child workflows, not from mutable macro fields alone.

---

## 1) Canonical Source of Truth

For Phase 1 onward, the workflow truth model is:

1. Child workflow lane state is authoritative.
2. Child workflow steps determine lane status.
3. Parent lifecycle stage is derived from lane status via handoff rules.
4. Legacy `transactions.current_main_stage` / `transactions.stage` are compatibility inputs for migration, not primary truth.
5. All blocker, ownership, and completion data is derived from workflow + evidence rules.

---

## 2) Parent Lifecycle (Canonical)

Official parent lifecycle stages (only these are terminal-stage states):

- `SETUP`
- `SALES_OTP`
- `FINANCE`
- `TRANSFER`
- `REGISTRATION`
- `COMPLETE`
- `CANCELLED`

### 2.1 Parent status enum

For any parent transaction:

- `WAITING`
- `IN_PROGRESS`
- `BLOCKED`
- `COMPLETE`
- `CANCELLED`
- `ON_HOLD`

Notes:
- `WAITING` = no next required lane is active yet.
- `BLOCKED` = at least one hard blocker exists.
- `ON_HOLD` = temporary user or legal hold not tied to workflow failure.

## 3) Child Workflow Lanes

Every transaction has one or more workflow lanes.

### 3.1 Core lanes (always considered)

- `sales_otp`
- `finance`
- `transfer`
- `registration`

### 3.2 Conditional finance/transfer lanes

- `finance_cash`
- `finance_bond`
- `finance_hybrid`
- `bond_registration`
- `seller_bond_cancellation`
- `attorney_transfer`

### 3.3 Effective lane mapping rules

- `finance` is a parent lane with finance-type children.
- `finance_cash`, `finance_bond`, and `finance_hybrid` are mutually exclusive by `financeType`, except where hybrid can expose both cash- and bond-like evidence requirements.
- `transfer` lane is active once finance handoff rules are met or a specific transfer exception is permitted by product policy.
- `bond_registration` is active when finance type is `bond` or `hybrid`.
- `seller_bond_cancellation` is active when seller has an existing bond.
- `attorney_transfer` is the legal handoff lane under `transfer` for conveyancing ownership actions.

## 4) Status Enums

### 4.1 WorkflowStatus (lane-level)

```ts
type WorkflowStatus =
  | 'not_started'
  | 'active'
  | 'blocked'
  | 'ready_for_handoff'
  | 'complete'
  | 'skipped'
  | 'cancelled';
```

- `not_started`: lane exists but no action taken.
- `active`: lane is in progress.
- `blocked`: lane cannot continue due to hard/soft blocker.
- `ready_for_handoff`: lane can be handed to next parent stage.
- `complete`: all required steps and required evidence complete.
- `skipped`: lane not applicable per rules.
- `cancelled`: transaction cancelled and lane superseded.

### 4.2 StepStatus (step-level)

```ts
type StepStatus =
  | 'not_started'
  | 'pending'
  | 'blocked'
  | 'complete'
  | 'skipped'
  | 'not_applicable';
```

- `pending`: dependencies are satisfied, waiting for action/document/signing.
- `blocked`: blocked by prerequisite or blocker.
- `not_applicable`: the step is intentionally excluded due to branch logic.

## 5) Blocker Model

Standard blocker object:

```ts
type WorkflowBlocker = {
  code: string;
  message: string;
  severity: 'hard' | 'soft';
  ownerRole: 'buyer' | 'seller' | 'agent' | 'attorney' | 'conveyancer' | 'bank' | 'system';
  workflowKey: string;
  stepKey?: string;
  requiredEvidence?: string[];
};
```

- A lane can have multiple blockers.
- At parent scope, if any lane has a hard blocker then parent status = `BLOCKED`.
- Soft blockers affect urgency/UX but do not enforce hard gate unless configured by rule.
- Each blocker should be generated by lane engines, not UI string concatenation.

## 7) Step Completion Draft Mapping

### 7.1 sales_otp

- `collect_buyer_details` ← evidence: `BUYER_ONBOARDING_COMPLETE`
- `collect_seller_details` ← evidence: `SELLER_ONBOARDING_COMPLETE`
- `collect_supporting_documents` ← evidence: `BUYER_FICA_COMPLETE`, `SELLER_FICA_COMPLETE`
- `generate_and_send_otp` ← evidence: OTP generated + portal links sent
- `sign_otp` ← evidence: `SIGNED_OTP_DOCUMENT`

### 7.2 finance_cash

- `collect_pof_or_cash_confirmation` ← evidence: `POF_DOCUMENT`
- `cash_review` ← manual owner confirmation

### 7.3 finance_bond

- `collect_bond_application` ← evidence: `BOND_APPLICATION_SUBMITTED`
- `bond_approval` ← evidence: `BOND_APPROVAL_LETTER`
- `guarantees` ← evidence: `GUARANTEE_ISSUED`

### 7.4 finance_hybrid

- `cash_portion_confirmation` ← evidence: `BANK_APPROVAL_LETTER` or `POF_DOCUMENT`
- `bond_portion` ← same gating as `finance_bond`

### 7.5 transfer (base / attorney transfer)

- `instruction_issued` ← attorney transfer kickoff
- `transfer_docs_prepared` ← `TRANSFER_DUTY_DA`
- `transfer_signing` ← signatures evidence
- `lodgement_package_ready` ← `TRANSFER_LODGEMENT_PROOF`

### 7.6 bond_registration

- `bond_reg_documents` ← supporting bond registration docs
- `bond_reg_confirmed` ← completion event from conveyancer/bank

### 7.7 seller_bond_cancellation

- `bond_cancellation_requested` ← seller has existing bond + cancellation request started
- `bond_cancellation_documents` ← `MORTGAGE_CANCELLATION_AUTH`

### 7.8 registration

- `ready_for_registration` ← transfer handoff reached
- `registration_confirmed` ← `FINAL_REGISTRATION_CERTIFICATE` or equivalent event

## 8) Handoff Rules (Core)

### 8.1 SETUP → SALES_OTP

- Trigger condition: transaction record exists with buyer/seller/property participants in scope and workspace active.
- Parent moves to `SALES_OTP` when `sales_otp` is `active` or any `sales_otp` step transitions out of `not_started`.

### 8.2 SALES_OTP → FINANCE

Finance may become `active` only if all required conditions are complete:
- `sales_otp.sign_otp` is `complete`.
- `sales_otp.collect_buyer_details` is `complete`.
- `sales_otp.collect_seller_details` is `complete`.
- `sales_otp.collect_supporting_documents` is `complete` (buyer + seller required proof per product rules).
- Optional: attorney approval gate if product config requires it.

If all true → `finance` becomes `active`, and parent can move to `FINANCE`.

### 8.3 FINANCE → TRANSFER

Transfer lane starts only when relevant finance lane is `ready_for_handoff`:
- Cash: `finance_cash` readiness.
- Bond: `finance_bond` readiness.
- Hybrid: both cash and bond portions ready.
- Parent transition to `TRANSFER` requires finance readiness plus transfer precondition checks.

### 8.4 TRANSFER → REGISTRATION

Registration becomes active only when transfer meets handoff conditions:
- `transfer` completion of required pre-lodgement steps.
- `attorney_transfer` and any `bond_registration`/`seller_bond_cancellation` active lanes at required state for handoff path.
- Property transfer docs/signatures evidence complete enough for registration dossier.

### 8.5 REGISTRATION → COMPLETE

- `registration.registration_confirmed` complete and immutable final registration evidence attached.
- Completion event must be auditable (who, when, what evidence).

### 8.6 Any stage → CANCELLED
- `CANCELLED` is only set by explicit cancel action/event.
- Child lane states become `cancelled` at termination, except already completed steps may remain for audit history.

## 9) Progress Calculation Principles (phase 2+)

Current stage-bucket progress (e.g., 20%, 40%) is **deprecated**.

New principle:
- Parent percentage is computed from weighted child step completion.
- Per-lane score = completed required steps / total required steps.
- Parent score = weighted aggregate across active lanes.
- Blocked steps still count as in-progress, not complete.
- Skipped lane steps do not dilute score.

Initial v1 weights:
- `sales_otp` 20%
- `finance` 30%
- `transfer` 30%
- `registration` 20%

## 10) Legacy Field Mapping (v1 Compatibility)

Legacy transaction values map into canonical parent stages only for compatibility/compat.

### 10.1 transactions.current_main_stage / stage mapping

| Legacy value              | Canonical parentStage |
|--------------------------|-----------------------|
| `AVAIL`                  | `SETUP`               |
| `RES` / `RESERVED`       | `SETUP`               |
| `OTP`                    | `SALES_OTP`           |
| `FIN`                    | `FINANCE`             |
| `ATTY`                   | `TRANSFER`            |
| `XFER` / `TRANSFER IN PROGRESS` | `TRANSFER`       |
| `REG` / `REGISTERED`     | `REGISTRATION`        |
| `COMPLETED` / `COMPLETE` | `COMPLETE`            |
| `CANCELLED` / `ARCHIVED` | `CANCELLED`           |

### 10.2 transaction.stage mapped fields

- `Proceed to Attorneys` / `Finance Pending` / `Transfer In Progress` / `Transfer Lodged` are treated as the corresponding canonical stage in `stage` → `parentStage` reconciliation only.
- Missing/invalid values do not block canonical calculation; they are normalized to `SETUP` with migration warning.

### 10.3 Migration rule

- Legacy values continue to be read for backward compatibility and migration, then ignored in favor of derived roll-up when available.

## 11) Ownership & Audit Rules

- `sales_otp` owner: agent + system with separate buyer/seller contribution roles.
- `finance` owner: agent + finance operator, with bank-facing evidence support.
- `transfer` owner: attorney/commercial conveyancer.
- `registration` owner: conveyancer/attorney.
- Every workflow step change writes immutable event record:
  - actor
  - timestamp
  - step key
  - old state → new state
  - evidence reference (if applicable)

## 12) Acceptance Criteria (Phase 2 Readiness)

Phase 1 is complete only when all criteria are agreed and documented:

1. Parent stage list is finalized.
2. Child lane list and conditional activation rules are finalized.
3. Lane and step status enums are fixed and versioned.
4. Evidence-to-step mapping is accepted for each major lane.
5. Blocker object shape is stable and used as source for UI messaging.
6. Handoff gate conditions are exact and non-overlapping.
7. Legacy field mapping is documented with a known migration path.
8. Progress rules are defined as step-based roll-up, not bucketed stage math.
9. Overview read-path is explicitly from roll-up service, with legacy values accepted only as migration input.
10. Roll-back/unknown-state behavior is defined for inconsistent child states.
